# Golden identity fixtures (T-52)

Эти фикстуры проверяют главное свойство продукта — вердикт, переживающий рескан
(ADR `roadmap/adr/0001-identity-and-verdict.md`) — через **реальный** пайплайн
`swb_cli.commands.enrich.enrich()` → upload → carry-over/diff, а не через
синтетические `swb_id`, вписанные вручную в тест (как делает
`tests/server/conftest.py::make_meta`). Тесты, которые их используют, лежат в
`tests/server/test_e2e_golden_identity.py`.

Каждая пара — это **два реальных снимка одного и того же проекта** (`v1` и
`v2`): настоящий SARIF-файл + настоящее дерево исходников на диске. Тест сам
вызывает `enrich()` на каждой паре (никакие `.swbmeta.json` здесь не лежат
предгенерированными — они не нужны на диске, `enrich()` дешёвый и
детерминированный, а хранение сгенерированных файлов означало бы держать их в
синхроне вручную при любой правке `cli/swb_cli/fingerprints.py`).

## Структура

```
golden/<scenario>/v1/report.sarif
golden/<scenario>/v1/repo/...        # дерево исходников для v1
golden/<scenario>/v2/report.sarif
golden/<scenario>/v2/repo/...        # дерево исходников для v2
```

В тесте `v1/repo` и `v2/repo` копируются в один и тот же по имени `repo_root`
(случайное имя на запуск теста, но одинаковое для `v1` и `v2` внутри одного
теста) — это ровно то, что делает `provenance.repo` в swbmeta стабильным между
двумя загрузками одного и того же тестового "проекта" на сервере
(`_build_provenance` в `cli/swb_cli/commands/enrich.py` берёт имя проекта из
`repo_root.name`).

## Сценарии

### `line_shift/` — сдвиг строк

`v2/repo/src/auth.py` — тот же файл, что и `v1`, с тремя вставленными строками
**выше** уязвимой строки (`query = ...`, было на строке 11, стало на строке 18
после вставки логгера). Содержимое самой находки не меняется.

Ожидание (ADR §4: content-hash без номера строки): `swb_id` совпадает между
`v1` и `v2` → вердикт, поставленный на находку в `v1`, переносится
(`carried`) на находку в `v2`; baseline-diff `v2` относительно `v1` — `unchanged`.

### `rename/` — переименование файла

`v1/repo/src/a.py` и `v2/repo/src/b.py` — байтово идентичный файл под разными
именами; SARIF `v1` указывает на `src/a.py`, `v2` — на `src/b.py`, та же строка.

Ожидание (ADR §1, строка 386 приложения A: перенос файла меняет `norm_uri` →
`swb_id` меняется на уровнях content/legacy): `swb_id` **не совпадает** между
`v1` и `v2` → вердикт **не переносится**, находка в `v2` остаётся `unmarked`;
baseline-diff — находка `v1` (`src/a.py`) уходит в `closed`, находка `v2`
(`src/b.py`) — в `new`, ни одна не попадает в `unchanged`.

### `reorder/` — перестановка результатов

Общее дерево исходников (`repo/src/one.py`, `two.py`, `three.py`, три разных
находки в трёх файлах) переиспользуется для `v1` и `v2` без изменений.
`v1/report.sarif` перечисляет находки в порядке `one → two → three`;
`v2/report.sarif` — те же три находки, порядок `three → one → two`; `ruleId`
каждого результата и позиция в `rules[]` тоже переставлены, чтобы перестановка
была настоящей, а не косметической.

Ожидание (ADR §2: occurrence считается после сортировки, не зависит от порядка
`results[]`): все три `swb_id` совпадают между `v1` и `v2` один-в-один →
каждая находка переносит свой вердикт (или остаётся `unmarked`, если вердикта
не было); baseline-diff — всё `unchanged`, `new`/`closed` пусты.

### `added_removed/` — находка исчезла, находка появилась

Общее дерево исходников (те же три файла, что и в `reorder/`).
`v1/report.sarif` = находки `one.py` (общая) + `two.py` (только в v1, дальше
"исчезает"); `v2/report.sarif` = находки `one.py` (общая) + `three.py`
(только в v2, "новая").

Ожидание: находка `one.py` переносит вердикт между `v1` и `v2` как обычно;
находка `two.py` остаётся как есть в `v1` (её `swb_id`/вердикт никуда не
делись — они просто не встречаются в `v2`) и попадает в `closed` при
baseline-diff `v2` относительно `v1`; находка `three.py` — новая, `unmarked`
в `v2`, попадает в `new` при diff.

## Как перегенерировать / добавить новую пару вручную

Предгенерированных `.swbmeta.json` нет — регенерировать нечего, они всегда
считаются заново из `report.sarif` + `repo/` при запуске теста реальным
`enrich()`. Чтобы добавить новый сценарий:

1. Создать `golden/<новый_сценарий>/v1/repo/...` и `.../v2/repo/...` —
   настоящие небольшие исходники (не more нескольких находок на файл, чтобы
   было легко проверить строки глазами).
2. Написать `golden/<новый_сценарий>/v1/report.sarif` и `.../v2/report.sarif`
   вручную (см. существующие файлы как шаблон — минимальный SARIF 2.1.0:
   `tool.driver.name`, `results[].ruleId/level/message/locations`). `startLine`
   должен реально совпадать со строкой находки в соответствующем файле —
   проверить `grep -n '<искомая строка>' golden/<сценарий>/v<N>/repo/<uri>`.
3. Проверить ожидание вручную перед тем, как писать assert'ы в тесте: вызвать
   `swb-cli enrich` (или `enrich()` напрямую, см.
   `tests/server/test_e2e_golden_identity.py::_enrich`) на обоих SARIF с
   `--repo-root` на соответствующее дерево и сравнить `findings[].swb_id`
   в двух получившихся `.swbmeta.json` — это ровно то, что тест будет
   проверять автоматически. Быстрый способ на хосте:
   ```
   swb-cli enrich golden/<сценарий>/v1/report.sarif \
     --repo-root golden/<сценарий>/v1/repo --out /tmp/v1.swbmeta.json
   swb-cli enrich golden/<сценарий>/v2/report.sarif \
     --repo-root golden/<сценарий>/v2/repo --out /tmp/v2.swbmeta.json
   python -c "import json; print(json.load(open('/tmp/v1.swbmeta.json'))['findings'][0]['swb_id'])"
   python -c "import json; print(json.load(open('/tmp/v2.swbmeta.json'))['findings'][0]['swb_id'])"
   ```
4. Добавить тестовую функцию(и) в `tests/server/test_e2e_golden_identity.py`,
   используя существующие хелперы `_enrich`/`_upload`/`_upload_pair`.

## Почему это ловит регрессию алгоритма, а не только серверную механику

`tests/server/conftest.py::make_meta` берёт `swb_id` из спецификации теста
(`spec.get("swb_id") or hash(...)`) — то есть carry-over/diff-тесты на этой
фабрике проверяют только "сервер верит переданному swb_id", но не сам
алгоритм в `cli/swb_cli/fingerprints.py`. Тесты в этой директории вызывают
настоящий `enrich()`, поэтому порча материала хеша, сортировки occurrence или
нормализации uri/window в `cli/swb_cli/fingerprints.py` меняет реальные
`swb_id`, и матрица красится — см. блок "мутационная самопроверка" в отчёте
задачи T-52 (`roadmap/done/T-52.md`).
